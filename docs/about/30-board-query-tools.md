# Board Query Tools

`src/server/assistant/board-query-tools.ts` provides two read-only tools that give the LLM visibility into board contents. These are prerequisites for any mutation — the LLM must query before it can update or delete.

## Why Read Tools Matter

Without query tools, the LLM has no way to know what's on a board. Chat history may contain stale references, and the model cannot "see" the canvas directly. These tools bridge the gap:

- **`summarize_board`**: structured overview for natural-language responses
- **`list_canvas_items`**: flat list of IDs for targeting mutations

The chat route enforces this pattern through runtime context instructions:

```
"When the user asks what is on the board, call summarize_board or list_canvas_items before answering."
"Board and item tool results are authoritative. Never invent item titles, counts, or content."
```

## `summarize_board` — Structured Board Overview

Returns a comprehensive summary suitable for generating natural-language responses.

**Input:** `{ boardId: string }`

**Output structure:**

```typescript
{
  boardId: string,
  boardTitle: string,
  boardDescription: string | null,
  totalItems: number,
  breakdown: [{ type: string, count: number, titles: string[] }],
  items: [{
    id: string,
    type: string,
    title: string | null,
    position: { x, y, width, height },
    content: {
      preview: string,    // 500-char trimmed summary
      title: string | null,
      text: string | null,
      href: string | null,
      alt: string | null,
      tasks: [] | null,    // Parsed task_list tasks
      columns: [] | null   // Parsed kanban columns
    }
  }]
}
```

**Execution:**

```typescript
const [board, items] = await Promise.all([
  getBoard(input.boardId),
  listCanvasItemsForBoard(input.boardId),
]);

if (!board || board.workspaceId !== context.workspaceId) {
  return { ok: false, summary: "Board not found." };
}
```

Both queries run in parallel via `Promise.all` for efficiency. Workspace ownership is verified against the board (all items on the board inherently share the board's workspace).

**Content summarization (`summarizeCanvasItemContent`):**

This function extracts readable information from each item's JSON content based on type:

| Item Type | Extracted Fields |
|---|---|
| sticky_note | title, text |
| text | title, text |
| markdown | title, text |
| image | src (as href), alt |
| link | href, title |
| task_list | tasks[] with titles and completed states |
| kanban | columns[] with names and card titles |

The preview is generated by joining extracted parts with ` | ` and truncating to 500 characters:

```typescript
const CONTENT_PREVIEW_LIMIT = 500;

function truncate(value: string, limit = CONTENT_PREVIEW_LIMIT): string {
  return value.length > limit ? `${value.slice(0, limit - 1)}...` : value;
}
```

Example preview: `Title: Sprint Review | Text: Review Q1 deliverables and plan next steps`

## `list_canvas_items` — Flat Item List for Targeting

Returns a flat array of items focused on IDs, positions, and raw content. This is the tool used before update/delete operations.

**Input:** `{ boardId: string }`

**Output:**

```typescript
[{
  id: string,
  type: string,
  x: number, y: number, width: number, height: number,
  content: Record<string, unknown>  // Raw JSON
}]
```

**Execution:**

Same board ownership check, then:

```typescript
const items = await listCanvasItemsForBoard(input.boardId);
const result = items.map((item) => ({
  id: item.id,
  type: item.type,
  x: Math.round(item.x),
  y: Math.round(item.y),
  width: Math.round(item.width),
  height: Math.round(item.height),
  content: item.content,
}));
```

Positions are rounded to integers for cleaner output. Content is passed as raw JSON — the LLM sees the full structured payload.

## Input Validation

Both tools share the same input shape and validator:

```typescript
function validateBoardIdInput(input: unknown): ToolValidationResult {
  if (!isObject(input)) return { ok: false, error: "Input must be an object." };
  if (typeof input.boardId !== "string" || !input.boardId.trim()) {
    return { ok: false, error: "boardId is required." };
  }
  return { ok: true };
}
```

## Registration & Permission

```typescript
export function registerBoardQueryTools(registry: ToolRegistry) {
  registry.register(summarizeBoardTool);
  registry.register(listCanvasItemsTool);
}
```

Both are permission level 1 — read-only, safe, no confirmation needed.

## JSON Schema for the LLM

Both tools expose the same schema:

```json
{
  "type": "object",
  "additionalProperties": false,
  "required": ["boardId"],
  "properties": {
    "boardId": { "type": "string" }
  }
}
```

## Query-Then-Mutate Flow

Consider this chat interaction:

```
User: "What's on this board?"
```

1. LLM calls `summarize_board({ boardId })` 
2. Receives: 5 items, 3 sticky_notes, 1 task_list, 1 kanban
3. Responds: "Your board has 5 items: 3 sticky notes titled X, Y, Z; a task list with 4 tasks; and a Kanban board with 3 columns."

```
User: "Delete the sticky note about Q1 planning"
```

4. LLM calls `list_canvas_items({ boardId })`
5. Scans items for "Q1 planning" in content titles
6. Calls `delete_canvas_item({ itemId: "xyz", confirmed: true })`
7. Tool succeeds → LLM responds: "I deleted the Q1 planning note."

The query tools ensure every mutation is informed by actual database state, not hallucinated information.

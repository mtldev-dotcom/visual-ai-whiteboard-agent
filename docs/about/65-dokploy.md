# Dokploy + Hetzner Deployment

This chapter summarizes the deployment guide at `docs/deployment/DOKPLOY_HETZNER.md`. It explains the Dokploy/Hetzner deployment strategy and why this combination was chosen for the project's first production-capable deployment target.

## What Are Dokploy and Hetzner?

### Dokploy

[Dokploy](https://dokploy.com) is a **self-hosted Platform as a Service (PaaS)** designed to deploy Docker containers. It is the closest open-source alternative to Heroku or Vercel, but runs on your own infrastructure.

Key capabilities:
- **Dockerfile-based deployments**: Build and run any Dockerfile from a connected Git repository.
- **Managed Postgres**: Create and manage PostgreSQL databases within Dokploy.
- **Built-in reverse proxy**: Handles TLS/SSL certificate provisioning and renewal automatically.
- **Git integration**: Connect GitHub repositories and trigger deployments on push.
- **Health checks**: Monitor container health and restart on failure.
- **Environment variable management**: Set and rotate environment variables per application.
- **Rollback**: One-click rollback to any previous deployment.
- **Web UI**: Manage everything through a dashboard.

### Hetzner

[Hetzner](https://www.hetzner.com) is a German cloud provider known for **affordable VPS hosting**. Their CX line of cloud servers provides:

- Competitive pricing (starting around €3-4/month for a basic VPS).
- European and US data center locations.
- Reliable network with DDoS protection.
- Block storage volumes for database persistence.
- Snapshots and backups.

Hetzner provides the raw compute, while Dokploy provides the platform layer on top.

### Why This Combination?

| Alternative | Dockploy + Hetzner beats it because... |
|---|---|
| Vercel | Not suitable for a container with Postgres + Prisma migrations + persistent file storage |
| Heroku | Free tier removed; paid plans expensive at scale |
| Railway | Good, but less control over networking and TLS |
| AWS ECS / EKS | Operational complexity exceeds what this project needs |
| Bare Docker on a VPS | Dokploy adds a management UI, rollback, health checks, and TLS without manual Nginx/Caddy config |

Dokploy on a €4-6/month Hetzner VPS provides a production-capable deployment with a management UI, automated TLS, and database orchestration — at a fraction of the cost of managed platforms.

## Pre-deployment Requirements

Before deploying, you need:

1. **A Hetzner VPS** with Dokploy installed. Dokploy provides a one-line install script:
   ```bash
   curl -sSL https://dokploy.com/install.sh | sh
   ```
   This installs Dokploy, Docker, and its dependencies.

2. **A domain or subdomain** pointing to the VPS IP address. TLS/SSL is handled by Dokploy's built-in reverse proxy (powered by Caddy), which requires a domain for automatic certificate provisioning.

3. **A PostgreSQL database** — either created inside Dokploy (managed Postgres service) or an external provider reachable from the VPS.

4. **GitHub repository access** — Dokploy pulls the repository to build the Dockerfile.

## Deployment Steps

### Step 1: Create the Database

In Dokploy's dashboard, create a PostgreSQL service:

- **Database name:** `visual_whiteboard_ai`
- **User:** Choose a strong username
- **Password:** Generate a strong password (store it securely)

After creation, Dokploy provides an **internal connection string** that the app container can use:

```
postgresql://USER:PASSWORD@HOST:5432/visual_whiteboard_ai
```

**Why use the internal connection string.** Dokploy places containers on the same Docker network. Using the internal hostname (generated by Dokploy) provides lower latency and avoids routing through the public internet. The `HOST` in this URL is a Docker service name, not a public IP.

**Do not use the `docker-compose.yml` connection string.** The compose file is for local development only and hardcodes dev credentials.

### Step 2: Create the Application

In Dokploy:

1. **Create a new application.**
2. **Connect the GitHub repository** (`your-org/visual-ai-whiteboard-agent`).
3. **Select Dockerfile deployment** (not Nixpacks). If Nixpacks is selected by default, switch to Dockerfile — Nixpacks may select Node.js 18, but the Dockerfile explicitly uses Node.js 22.
4. **Dockerfile path:** `Dockerfile` (in the repository root).
5. **Exposed port:** `3000`.
6. **Health check path:** `/api/health`.

### Step 3: Set Environment Variables

Enter these variables in the Dokploy app environment settings:

```bash
NODE_ENV=production
PORT=3000
DATABASE_URL=postgresql://USER:PASSWORD@HOST:5432/visual_whiteboard_ai
AUTH_SECRET=<generate-with-openssl-rand-base64-32>
NEXTAUTH_URL=https://your-domain.example
APP_URL=https://your-domain.example
APP_SIGNUP=enable
OPENROUTER_API_KEY=<your-openrouter-api-key>
OPENROUTER_MODEL=anthropic/claude-3-haiku
```

Optional variables (add when ready):

```bash
TELEGRAM_BOT_TOKEN=<botfather-token>
TELEGRAM_WEBHOOK_SECRET=<random-secret>
```

### Step 4: Deploy

Trigger the initial deploy. Dokploy will:

1. Clone the repository.
2. Run the multi-stage Docker build (described in Chapter 64).
3. Generate the Prisma client in the `deps` stage.
4. Build the Next.js application in the `builder` stage.
5. Assemble the runtime image with `.next`, Prisma, and `docs/agent-core`.
6. Start the container — which runs `prisma migrate deploy` followed by `next start -H 0.0.0.0`.

**Expected startup time:** 2-4 minutes for the first deploy (build + migration). Subsequent deploys use cached Docker layers and are faster.

### Step 5: Verify

Open the health check endpoint:

```
https://your-domain.example/api/health
```

Expected response:

```json
{
  "ok": true,
  "service": "visual-ai-whiteboard-agent"
}
```

Then test the full user flow:

1. **Sign up** at `https://your-domain.example/signup`.
2. **Confirm** you land on the home page after signup.
3. **Verify** the Welcome Board is created automatically.
4. **Ask the assistant** to add a sticky note ("Add a sticky note saying Hello World").
5. **Refresh** and confirm the board and chat persist.
6. **Ask the assistant** what it can see on the board — confirm it describes actual canvas items.
7. **Delete the note** via the assistant — confirm it disappears.

After the initial account exists, optionally set `APP_SIGNUP=disable`, redeploy/restart, and confirm:
- `/signup` shows a disabled page.
- `POST /api/auth/signup` returns 403.
- `/login` still works for existing accounts.

## Health Check Architecture

The health check endpoint (`/api/health`) is critical for Dokploy's container monitoring:

```ts
// src/app/api/health/route.ts
export async function GET() {
  return Response.json({
    ok: true,
    service: "visual-ai-whiteboard-agent",
  });
}
```

**What the health check does:**
- Confirms the Next.js app is serving requests.
- Confirms routing is functional.

**What the health check does NOT verify:**
- Database connectivity (that requires a dedicated endpoint or `db:smoke`).
- LLM availability (OpenRouter may have transient downtime).
- Auth flow (requires a full login cycle).

**Why a simple health check is sufficient for Dokploy.** Dokploy checks the endpoint periodically and restarts the container if it fails. A simple HTTP 200 response confirms the app process is alive. More sophisticated checks (database, LLM) would add latency and could trigger unnecessary restarts for transient downstream issues.

If database connectivity verification is needed at container startup, it happens during `prisma migrate deploy`, which runs before `next start`. If migrations fail, the container exits, and Dokploy will report the failure.

## Auto-Deploy on Push

Dokploy supports automatic deployments when the connected Git branch receives a push. Enable this if you want continuous deployment:

1. In Dokploy, go to the application settings.
2. Enable "Auto Deploy."
3. Specify the branch (e.g., `main`).

**When to enable:**
- **Enable** when the branch is stable and you want immediate deployment of merged changes.
- **Disable** when you want to manually review changes before deploying (recommended for initial setup and during active development).

## Rollback

Dokploy keeps a history of successful deployments. To rollback:

1. In Dokploy, go to Deployments for the application.
2. Find the previous successful deployment.
3. Click "Rollback."

**Caveats about rollback:**

- **Database migrations are forward-only.** Rolling back the application code does not roll back applied database migrations. If a migration added a column, the old code will simply ignore it (Prisma queries are selective). If a migration removed a column, the old code may fail.
- **If a migration removal is required,** create a new migration that restores the removed column rather than manually editing the production database.
- **Never directly modify the production database.** Use Prisma migrations for all schema changes.

## Optional: Telegram Webhook Setup

After the app is reachable over HTTPS:

1. Set `TELEGRAM_BOT_TOKEN` and `TELEGRAM_WEBHOOK_SECRET` in the Dokploy environment.
2. Redeploy or restart the container.
3. Run the webhook registration script:

```bash
# In the Dokploy container shell or as a one-off command
npm run telegram:webhook
```

This registers the webhook URL with Telegram:

```
POST https://api.telegram.org/bot<TOKEN>/setWebhook?url=https://your-domain.example/api/telegram/webhook
```

After registration, Telegram sends message updates to the webhook endpoint. The app processes commands (`/boards`, `/tasks`, `/newboard`, `/addnote`) and replies through Telegram's `sendMessage` API.

## TLS and Security

Dokploy handles TLS through its built-in reverse proxy (powered by Caddy). This means:

- **No manual certificate management.** Certificates are provisioned automatically via Let's Encrypt.
- **Auto-renewal.** Certificates renew before expiration without intervention.
- **HTTPS by default.** HTTP requests are redirected to HTTPS.

**Additional security considerations for production:**

| Concern | Mitigation |
|---|---|
| `AUTH_SECRET` weak or default | Generate a strong random secret with `openssl rand -base64 32` |
| Signup open to public | Set `APP_SIGNUP=disable` after creating initial accounts |
| Database exposed to internet | Use Dokploy's internal network — do not publish the database port |
| Environment variable leaks | Never commit `.env` files; use Dokploy's encrypted secret storage |
| Container running as root | The Dockerfile does not set a non-root user — add `USER node` for production hardening |

## Current Caveats

As noted in `docs/deployment/DOKPLOY_HETZNER.md`, the following are not yet configured for production:

- **No production file storage** — canvas items currently store data in Postgres (JSON fields). File uploads (images, attachments) are not implemented.
- **Credentials auth only** — OAuth and magic-link authentication are P2 backlog items.
- **No E2E tests** — the deployment is verified through manual QA, not automated browser tests.
- **Production hardening** — the Dockerfile runs as root, has no resource limits, and lacks monitoring/alerting.
- **Telegram feature gaps** — `/remind`, `/summarize`, file capture, and voice transcription are pending P1 tasks.

## When to Use This Deployment

This deployment path is suitable for:

- **Test deployments**: verify the app works in a production-like environment.
- **Internal team use**: small team with a few concurrent users.
- **Demonstrations**: show the app to stakeholders with a real domain and database.

It is **not yet suitable** for:

- **Public multi-tenant SaaS** (needs auth hardening, rate limiting, scaling, and monitoring).
- **High-availability production** (needs database replication, load balancing, and failover).
- **Compliance-regulated environments** (needs audit logging, backup verification, and security review).
